# 当前网站架构解析与性能优化建议

## 1. 当前网站架构

结合当前项目代码和你提供的线上部署描述，当前网站架构可以概括为：

### 1.1 部署组成

- `ECS` 服务器：
  - 部署 `JavaSB_back` Spring Boot 后端
  - 部署 `Backend_front` 后台前端静态资源
  - 使用反向代理把后台和小程序请求转发到后端
- `MySQL` / `Redis`：
  - 由后端直接访问
- `OSS`：
  - 文件上传落地到 OSS
- `CDN`：
  - 资源通过 CDN 下发
- `DNS`：
  - 域名解析到 ECS / CDN

### 1.2 请求流向

#### 后台前端

1. 浏览器访问后台域名
2. Nginx/反代返回 `Backend_front/dist`
3. 浏览器请求 `/api/...`
4. 反向代理转发到 `JavaSB_back`
5. 后端访问 MySQL / Redis / OSS / OCR 等服务
6. 资源 URL 返回后，浏览器再从 CDN 拉取图片、试卷、视频等静态资源

#### 小程序

1. 小程序访问后端接口域名
2. 请求进入 ECS 反向代理
3. 反向代理转发到 Spring Boot
4. 后端返回业务 JSON
5. 试卷图片、切片图片、课程封面、视频等资源从 CDN 拉取

### 1.3 文件流向

#### 当前实际数据流

- 上传：
  - 后台普通文件上传很多仍然是“先传到后端，再由后端传 OSS”
  - 部分场景已有 OSS 直传能力，例如课程视频 `upload-video-signature`
- 下载：
  - 资源 URL 大多是 CDN 地址
  - 后端通过 `OssStorageService.toCdnUrl()` 把资源转为 CDN 可访问地址

### 1.4 当前架构图

```text
用户浏览器 / 微信小程序
        |
        v
  域名 + DNS 解析
        |
        v
   Nginx / 反向代理
   |              |
   |              +--> 后台静态资源 dist
   |
   +--> Spring Boot 后端
            |
            +--> MySQL / Redis
            +--> OSS
            +--> OCR / AI / 短信 / 支付
                    |
                    +--> CDN 分发静态资源
```

## 2. 当前架构下的主要问题

你反馈的 3 个问题分别是：

1. 后台第一次进入很慢，首屏加载接近 10 秒。
2. 后台切换页面有卡顿感。
3. 试卷编辑页上传试卷后，页面没有立刻显示“已上传”；AI 识别结果也不是立刻可见，往往要等一会并刷新重新进入才显示。

这些问题不是单点故障，而是“前端打包体积 + 反代链路 + 后端同步重任务 + 页面状态同步策略”共同造成的。

## 3. 代码层已发现的关键瓶颈

## 3.1 后台首屏慢的核心原因：前端产物体积过大

根据实际构建结果，后台前端打包后存在超大 chunk：

- 主 chunk 约 `1.39 MB`
- `course-dialog` 约 `820 KB`
- `install` 约 `764 KB`
- `PaperRegionEditor` 约 `326 KB`

这说明当前后台存在明显问题：

- 首屏加载 JS 体积过大
- 解析和执行 JS 耗时高
- 首次进入不仅慢在网络，也慢在浏览器主线程执行

### 代码迹象

`Backend_front/vite.config.ts` 中：

- 没有做 `manualChunks` 精细拆包
- `chunkSizeWarningLimit` 被放大到 `2000`
- `vite-plugin-vue-devtools` 在生产配置中也被启用

这意味着当前更像是“允许大包存在”，而不是主动优化大包。

## 3.2 页面切换卡顿：路由切换时加载和渲染成本太高

页面切换卡顿通常来自 3 类问题：

### 1. 切页时要拉大体积 chunk

像考试编辑、试卷编辑、富文本、图表、视频等页面都偏重，一旦懒加载 chunk 很大，路由切换会明显卡顿。

### 2. 页面组件本身渲染重

例如试卷编辑页：

- 图片/长图渲染
- 选框编辑
- PDF iframe
- 学生列表
- OCR 状态轮询

这些都在单页里堆叠，切到该页面时渲染成本很高。

### 3. 页面切换带有额外请求

例如 `PaperEditView.vue` 中会同时做：

- `loadConfig()`
- `loadStudents()`

而且很多时候用户只是编辑“模板/原卷”，其实不需要一进来就拉“考生原卷列表”。

这会增加切页时的请求数和等待时间。

## 3.3 试卷上传后不能立刻显示：后端链路是同步重处理

这是当前最明确的问题之一。

在 `ExamProjectServiceImpl.java` 中，上传公共试卷并不是“纯上传即返回”，而是包含了同步重处理逻辑：

### 上传 PDF 时当前流程

1. 前端上传文件到后端
2. 后端读入整份 PDF
3. 使用 `PDFBox` 逐页渲染
4. 把所有页面拼成一张长图
5. 生成分页位置信息
6. 再上传到 OSS
7. 再保存数据库记录

对应代码链：

- `storePaperAsset()`
- `renderPdfAsLongImage()`
- `derivePaperMergeInfo()`
- `savePaperMergeInfo()`

这意味着：

- 上传接口不是轻量请求
- PDF 越大，响应越慢
- 反向代理如果超时或用户网络稍差，就会出现“文件实际已经处理完，但前端当前请求没拿到最终状态”

## 3.4 试卷编辑页前端状态同步机制不够稳定

`PaperEditView.vue` 里虽然已经做了补救：

- `refreshConfigSilently()`
- `confirmPaperUploadResult()`
- `confirmAutoCutResult()`

但它仍然存在几个问题：

### 1. 上传完成状态依赖轮询确认

如果后端处理慢，前端要靠轮询再确认，而不是后端明确返回“已完成且可展示”的最终状态。

### 2. 状态语义不一致

后台科目汇总页里，“试卷录入”使用的是：

- `paperCount / studentCount`

这个更像“学生答卷录入进度”，不是“原卷/模板是否上传成功”。

所以你在“试卷编辑”里上传了公共试卷后，外层汇总页仍然可能显示“未录入”，造成错觉。

### 3. AI 识别结果保存后，外层页面没有单独展示“布局已生成”

现在外层页面没有“布局状态”字段，用户很难判断：

- 文件是否已上传
- 分页信息是否已生成
- OCR 布局是否已保存

最终体验就是“我明明操作成功了，但页面看起来像没成功”。

## 3.5 反向代理和单机部署会放大卡顿

当前是：

- ECS 同时跑后端
- ECS 同时承载后台前端静态资源
- 还要处理试卷上传、OCR、PDF 渲染、数据库读写、CDN 回源等请求

如果 ECS 规格一般，容易出现：

- CPU 被 PDF 渲染 / OCR / Java GC 占用
- 后台静态资源响应抖动
- API 响应时间波动
- 首屏和页面切换变慢

## 4. 三个问题的可能根因拆解

## 4.1 后台第一次进入很慢

### 高概率原因

1. 后台前端打包体积过大，JS 下载和执行慢。
2. 主入口 chunk 太大，浏览器解析执行时间长。
3. 首页默认页面本身组件重、接口多、图表多。
4. ECS 资源有限，后端与静态资源共享机器资源。
5. gzip 虽已开启，但没有更细的 chunk 拆分，导致首屏仍旧过重。

## 4.2 切换页面卡顿

### 高概率原因

1. 路由级 chunk 体积仍然偏大。
2. 某些页面切换时请求太多，且并发同步触发。
3. 重页面如试卷编辑器、图表页、复杂弹窗页渲染开销高。
4. 页面切换时主线程被大组件初始化占满。

### 更进一步判断：这是前端问题还是后端问题？

结论是：

- **切换页面“不流畅、发涩、卡顿感强”主要是前端问题**
- **如果切页后页面内容长时间空白、表格迟迟不出数据，则是后端接口耗时也在放大问题**

也就是说，这不是“纯前端”或“纯后端”的二选一，而是：

- **前端负责决定切页动画、组件挂载、JS 执行和页面响应是否丝滑**
- **后端负责决定切页后数据什么时候回来**

从当前项目代码来看，**主因偏前端，后端是次级放大器**。

## 4.2.1 前端侧导致切页卡顿的具体原因

### 1. 路由切换链路本身比较重

当前后台路由切换不是简单的 `router.push()`，而是叠加了多层处理：

- `beforeEach` 路由守卫
- `NProgress`
- 动态路由校验 / 首次路由注册
- `worktab` 标签页处理
- `Transition`
- `KeepAlive`
- `Teleport` 遮罩

相关代码：

- `Backend_front/src/router/guards/beforeEach.ts`
- `Backend_front/src/router/guards/afterEach.ts`
- `Backend_front/src/components/core/layouts/art-page-content/index.vue`
- `Backend_front/src/store/modules/worktab.ts`

这些机制单独看都没问题，但叠加在一起后：

- 页面切换成本变高
- 路由守卫逻辑变长
- 动画和缓存切换更容易放大“卡顿体感”

### 2. 页面切换使用了双层 `Transition`

在 `art-page-content/index.vue` 中：

- 缓存页面走一层 `Transition + KeepAlive`
- 非缓存页面又走一层 `Transition`

并且使用了：

- `mode="out-in"`
- `appear`

这会带来一个现象：

- 旧页面先退场
- 新页面再进场
- 如果新页面本身加载慢，就会明显感知到“停一下再出来”

如果切到重页面，比如：

- 试卷编辑页
- 分析页
- 带图表和大表格的页面

这种 `out-in` 过渡会让用户更容易感知“顿一下”。

### 3. `KeepAlive` + 标签页管理会增加切换复杂度

当前页面容器里用了：

```vue
<KeepAlive :max="10" :exclude="keepAliveExclude">
```

同时 `worktab` store 里又在维护：

- `opened`
- `keepAliveExclude`
- 动态标签复用
- 关闭时缓存排除

这意味着切页时除了页面渲染，还要处理：

- 标签状态同步
- 缓存命中/失效
- 当前标签激活

如果页面组件本身很重，缓存管理又复杂，就更容易在切页时产生“阻滞感”。

### 4. 首次进入某些页面时，chunk 太大

根据实际构建产物，后台存在多个大 chunk：

- 主入口约 `1.39MB`
- `PaperEditView` 所在链路依赖的编辑器 chunk 很大
- `PaperRegionEditor` 本身也很重

这意味着：

- 切换到某些页面时，不只是执行页面逻辑
- 还要先下载 / 解析 / 执行一整块 chunk

因此会出现：

- 页面点击后有明显延迟
- 切页时鼠标已点击，但 UI 过一会才进入

### 5. 试卷编辑页切换时额外请求过多

在 `PaperEditView.vue` 中：

- `loadConfig()`
- `loadStudents()`

都会在切换 tab 或进入页面时触发。

问题在于：

- 用户进入 `template/original` 编辑，本质上只需要试卷配置
- 但页面往往还会同时拉“考生原卷列表”

这会带来：

- 多一个接口等待
- 列表渲染负担
- 进入页面更重

### 6. 全屏遮罩和过渡掩码增加了“体感迟滞”

`art-page-content/index.vue` 中有：

- `showTransitionMask`
- 页面切换时显示一个全屏遮罩

虽然它本意是“提升视觉体验”，但在性能不够时会产生反效果：

- 用户会觉得页面“被挡了一层”
- 即使切页只慢 200~400ms，主观感觉也会更明显

## 4.2.2 后端侧如何放大切页卡顿

虽然后端不是切页卡顿的主因，但它会在下面几类页面明显放大问题：

### 1. 切页后立刻请求重接口

例如考试项目页、分析页、试卷编辑页，会在页面 mounted 后立即请求：

- 项目详情
- 成绩汇总
- 学生列表
- 试卷配置

如果这些接口响应慢，前端切页就会表现成：

- 页壳子先出来
- 内容慢慢补
- 用户体感是“切过去了，但没准备好”

### 2. 某些接口背后带有重计算

特别是试卷编辑相关接口，后端可能会做：

- OSS 下载
- PDF 转长图
- mergeInfo 推导
- OCR 预处理

这些接口一旦出现在切页后的首批请求里，就会让切页体验进一步变差。

## 4.2.3 结论：切页卡顿主要是谁的问题？

### 结论排序

1. **第一责任人：前端路由切换和页面资源体积**
2. **第二责任人：页面初始化时并发请求过多**
3. **第三责任人：后端部分接口耗时高**

所以优化顺序也应该是：

1. 先减轻前端切页负担
2. 再减少页面初始请求数
3. 最后优化后端接口耗时

## 4.2.4 针对切页卡顿的修正方案

## A. 前端优先优化

### 1. 降低路由切换动画复杂度

建议：

- 对重页面关闭 `out-in`
- 或对重页面直接使用 `fade`
- 或在系统设置里默认关闭页面切换动画

优先级很高，因为它能直接改善“主观卡顿感”。

### 2. 对重页面关闭 `appear`

首次进入页面不需要过渡动画时，可以禁用 `appear`，减少首次切页动画和重渲染叠加。

### 3. 对重页面单独禁用 `KeepAlive`

不是所有页面都适合缓存。

特别是这些页面建议谨慎缓存：

- 试卷编辑页
- 大图表页
- 大表格页
- 文件预览页

因为这类页面：

- 占内存大
- 恢复成本也高
- 容易和 `worktab` 的缓存排除逻辑互相影响

### 4. 按页面拆包，不只按路由拆包

重点把以下模块单独分包：

- `PaperRegionEditor`
- `PaperEditView`
- 图表模块
- Excel 导入模块
- 富文本编辑模块

### 5. `PaperEditView` 改为真正按 tab 懒加载

建议改成：

- 页面进入时只调用 `loadConfig()`
- `activeTab === 'student'` 时再调用 `loadStudents()`

不要在 `template/original` 编辑场景下提前拉学生列表。

### 6. 对页面切换加“预取”

例如用户从项目汇总点“试卷分析”按钮时：

- 先预取对应 route chunk
- 再导航

这样点击后会顺很多。

## B. 后端配合优化

### 1. 把页面首批请求按轻重拆开

例如项目编辑页：

- 首屏先返回基本信息
- 成绩汇总和学生列表再异步补

不要一上来就让页面等待全部数据齐全。

### 2. 减少切页触发的重逻辑

试卷配置接口如果发现：

- 还要下载 OSS
- 还要推导 mergeInfo
- 还要生成 preview

那这些动作应该提前做，不应在用户切页时再做。

### 3. 为关键接口记录耗时

建议对这些接口打耗时日志：

- `/api/system/exam-project/detail/*`
- `/api/system/exam-project/scores/summary`
- `/api/system/exam-project/scores/list`
- `/api/system/exam-project/papers/config`

这样能判断切页慢到底慢在：

- 路由切换
- JS 执行
- 还是 API 响应

## 4.2.5 最终判断

如果你问“切换页面不流畅到底是前端问题还是后端问题”，我的判断是：

- **体感卡顿，主要是前端问题**
- **内容慢出来，后端也有责任**

更准确一点说：

- **切页发涩、切页阻滞、点击后不立即响应：前端主因**
- **切过去后数据空白、要等几秒表格才出来：后端接口主因**

所以要想真正把“切页卡顿”解决掉，最有效的不是只优化后端，而是：

1. 先处理前端切页链路和重页面拆包
2. 再把页面初始化请求数压下来
3. 再优化后端接口耗时

这样效果会最明显。

## 4.3 上传试卷后未立刻显示

### 高概率原因

1. 后端上传接口里做了同步 PDF 转长图处理，耗时长。
2. 前端上传完成后虽然有刷新确认，但仍不是强一致的“处理完成回执”。
3. 外层页面展示的是“学生试卷录入状态”，不是“公共试卷上传状态”，导致状态含义错位。
4. AI 识别也是同步重任务，请求长、页面状态更新链长，所以用户感知为“没立刻生效”。

## 5. 优化建议

下面按优先级分层给建议。

## 5.1 P0：先解决用户最明显的问题

## A. 把试卷上传和 PDF 转长图改成异步任务

### 当前问题

上传接口里直接做：

- PDF 渲染
- 长图拼接
- 分页信息生成
- OSS 上传

这会导致接口响应慢。

### 建议方案

把流程改成：

1. 上传原始文件到 OSS
2. 数据库先记录状态为 `PROCESSING`
3. 后台异步任务处理 PDF 转图 / mergeInfo / 预览图
4. 完成后更新状态为 `READY`
5. 前端轮询任务状态或用 WebSocket / SSE 接收完成通知

### 推荐增加的字段

在 `exam_subjects` 中增加：

- `original_status`
- `template_status`
- `original_process_message`
- `template_process_message`
- `original_preview_url`
- `template_preview_url`
- `original_updated_at`
- `template_updated_at`

状态建议：

- `NOT_UPLOADED`
- `UPLOADING`
- `PROCESSING`
- `READY`
- `FAILED`

### 直接收益

- 上传接口秒级返回
- 用户不会误以为上传失败
- 前端可以显示“处理中”
- 失败时可以明确展示失败原因

## B. 外层页面状态文案要改成真实语义

当前“试卷录入”更接近“学生答卷录入进度”，不等于“原卷/模板已上传”。

### 建议

在项目科目汇总接口中补充字段：

- `originalUploaded`
- `templateUploaded`
- `originalReady`
- `templateReady`
- `layoutReady`

前端显示拆开：

1. 公共原卷
2. 模板答题卡
3. 考生答卷进度
4. AI 布局进度

这样用户就不会把“公共试卷上传”误解成“学生试卷已全部录入”。

## C. 试卷编辑页不要默认并行加载全部学生列表

当前 `PaperEditView.vue` 中，进入页面时就可能同时做：

- `loadConfig()`
- `loadStudents()`

### 建议

改为按 tab 懒加载：

- 打开 `template/original` 时，只加载配置
- 切到 `student` 时，再加载学生列表

### 直接收益

- 进入试卷编辑页更快
- 页面切换更顺滑
- 减少无意义 API 请求

## 5.2 P1：优化后台首屏和切页速度

## A. 对后台前端做 `manualChunks` 拆包

### 当前问题

构建产物中已有多个超大 chunk。

### 建议拆分策略

1. `vue` / `vue-router` / `pinia`
2. `element-plus`
3. `echarts`
4. `xlsx`
5. `wangeditor`
6. `paper-editor` 相关组件单独拆包
7. `course-dialog` 这类大弹窗页面单独 chunk

### 目标

- 首屏 chunk 尽量压到 `300KB ~ 500KB` 以内
- 重组件按路由和功能延迟加载

## B. 生产环境关闭 `vite-plugin-vue-devtools`

当前 `vite.config.ts` 中开发工具插件没有明显按环境关闭。

### 建议

只在 `development` 启用：

```ts
mode === 'development' ? vueDevTools() : null
```

## C. 为首屏页面做“轻首页”

如果后台默认首页是：

- 仪表盘
- 图表多
- 卡片多
- 请求多

建议把默认首页改成轻量版：

- 只放核心数据
- 图表延迟加载
- 非首屏模块滚动后再加载

## D. 页面切换中的大组件按需挂载

例如试卷编辑器、图表、复杂弹窗：

- 只在需要时 `defineAsyncComponent`
- 弹窗打开时再拉组件
- 避免页面初始就注册重组件

## E. 路由切换增加预加载和骨架屏

切换卡顿不一定完全能消掉，但可以改善感知：

- 常用页面做 hover / click 前预加载
- 页面切换先展示 skeleton
- 表格区域先占位，接口返回后再替换

## 5.3 P1：优化后端处理链路

## A. OCR 相关任务不要同步阻塞业务接口

AI 识别现在仍属于重任务，建议改为任务型接口：

1. 提交 OCR 任务
2. 返回 `taskId`
3. 后台异步执行
4. 前端轮询结果

这样可以避免：

- 长请求超时
- 页面一直 loading
- 用户误判“没有生效”

## B. 减少重复 OSS 下载和重复 PDF 渲染

当前链路中存在一些兜底逻辑，会在读取时再次：

- 下载 OSS 文件
- 渲染 PDF
- 生成 preview

### 建议

只在上传完成时生成一次：

- 长图预览
- mergeInfo
- 缩略图 / 预览图

后续读取直接使用数据库中已有结果，不再重复计算。

## C. 试卷处理服务单独线程池

如果仍保留 Java 内异步处理，建议至少：

- 使用独立线程池处理 PDF / OCR 任务
- 不占用 Tomcat 请求线程

## 5.4 P2：基础设施层优化

## A. 静态资源尽量让 CDN 直出

后台前端静态资源建议：

- ECS 上 Nginx 开启强缓存
- 静态资源尽量通过 CDN 分发

避免每次首屏都直接打 ECS。

## B. ECS 与重任务解耦

如果访问量和文件处理量上来，建议拆分：

1. `Web/API ECS`
2. `异步任务 Worker`

让：

- API 服务专心处理请求
- PDF 转图 / OCR / 试卷切片由 Worker 跑

## C. 监控必须补齐

建议至少接入：

- Nginx access log / upstream 响应时间
- Spring Boot 接口耗时统计
- MySQL 慢查询
- JVM CPU / 堆内存 / GC 时间
- OSS / OCR 外部调用耗时

没有监控时，只能凭感觉优化，很难定位“慢在前端、慢在后端、还是慢在外部服务”。

## 6. 建议的优化执行顺序

### 第一阶段：1 到 3 天内可做

1. 试卷上传改成“上传完成 + 后台异步处理”。
2. 项目科目汇总接口增加真实状态字段，修正前端状态文案。
3. `PaperEditView` 改成按 tab 懒加载学生列表。
4. 关闭生产环境 `vueDevTools`。
5. 对后台前端做 `manualChunks` 拆包。

### 第二阶段：3 到 7 天内可做

1. OCR 和 PDF 转图改为任务队列。
2. 大页面和大弹窗异步组件化。
3. 后台默认首页做轻量化改造。
4. 增加接口耗时日志和前端性能埋点。

### 第三阶段：中期优化

1. Web/API 与重处理 Worker 拆开部署。
2. 静态资源前移到 CDN。
3. 引入统一任务中心和状态通知机制。

## 7. 针对你当前 3 个问题的直接结论

## 问题 1：后台第一次进入慢

### 直接结论

最主要是后台前端首屏资源太大，不是单纯网络问题。

### 最优先动作

- 拆包
- 关生产 devtools
- 轻首页

## 问题 2：后台切页卡顿

### 直接结论

主要是重页面 chunk 大、组件初始化重、切页时请求过多。

### 最优先动作

- 路由级拆包
- 弹窗/编辑器异步加载
- 按 tab 懒加载

## 问题 3：试卷上传后不立刻显示

### 直接结论

最主要是“同步重处理 + 前端状态语义不清 + 页面状态更新链太长”。

### 最优先动作

- 上传改成异步任务
- 给试卷处理增加明确状态字段
- 前端把“处理中 / 已完成 / 失败”显示出来

## 8. 最终建议

当前网站架构本身没有方向性错误，属于典型的：

- ECS 承载业务接口
- 反向代理统一入口
- OSS 存储
- CDN 分发
- 小程序和后台共用后端

问题主要出在：

1. 前端后台包太大，首屏和切页变慢。
2. 后端把 PDF 转图、试卷预处理、OCR 等重任务放在同步请求里执行。
3. 页面状态没有精细化表达，导致用户感知“没成功”“没刷新”。

所以正确的优化方向不是推翻现有架构，而是：

- 保留现有部署架构
- 对前端做拆包和懒加载
- 对后端做异步任务化
- 对状态流做可视化和强一致回执

这样既能保住当前系统可用性，也能把首屏速度、切页体验和试卷编辑回显问题一起解决。
